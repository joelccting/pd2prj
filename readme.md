 底層實作
========================

【前端】

1. 介紹
    包含什麼：校園地圖的視覺化界面，包括地圖渲染、UI 元件、使用者互動功能
    功能：使用者可以查看地圖、選擇起點與目的地（支援多目的地）、設定路線偏好與交通工具、查看即時陰影和樹木覆蓋、透過 God Mode 編輯路網
    怎麼使用：在瀏覽器中打開 index.html，與地圖互動選擇路線、點擊「開始巡航導航」獲取結果；啟動前需先執行後端 Python 伺服器（http://localhost:8000）

2. 方法論

    技術棧：Leaflet.js（地圖庫）、原生 HTML/CSS/JavaScript（無框架）
    流程：
        載入 OpenStreetMap 地圖瓦片，並限制地圖範圍於校園邊界
        啟動時向後端 GET /api/graph 取得節點與邊的圖數據，建立前端本地 Graph 實例
        將節點渲染為分類彩色標記（學校/餐飲/住宿/其他），點擊可查看建築資訊
        監聽使用者在 UI 面板的選擇：起點、多個目的地、路線偏好、交通工具、巡航模式
        點擊「開始巡航導航」後，依巡航模式向後端 POST /api/route 請求路徑，前端負責多段路徑的串接與 TSP 近鄰法排序
        取得節點序列後，從本地 graph.adjacencyList 查詢邊距離，在前端計算預估旅行時間（含坡度修正、各交通工具速度、牽車路段）
        使用 L.polyline 繪製路徑，分色標示正常路段（藍/深綠）與須牽車路段（橘色虛線），並以 leaflet-polylineDecorator 加上方向箭頭
        林蔭模式下可疊加即時陰影圖層，依邊的 tree_shade / building_shade 屬性篩選並繪製
    語言：原生 JavaScript（ES Module，僅 app.js 使用 import）
    實現細節：
        L.map 初始化地圖，設定校園經緯度邊界與縮放級別（16–22）
        L.tileLayer 加載 OpenStreetMap 圖層（512px tile + zoomOffset）
        L.divIcon + CSS 自訂彩色圓點標記表示各分類建築
        前端維護 graph（Graph 實例）與 globalData（原始 JSON），路徑繪製和時間計算皆查詢這兩份資料
        多目的地「自動優化」採前端貪婪近鄰 TSP（以直線距離排序），不依賴後端排序
        搜尋框支援即時篩選，下拉選單依分類兩段式展開（先選分類、再選建築）
        面板支援收合/展開，語言可切換繁中/英文（lang.js 管理翻譯）
        God Mode（editor.js）允許在地圖上直接新增/刪除節點與邊，並可匯出修改後的路網 JSON
3. 相關檔案
   - js/app.js主應用邏輯：地圖初始化、API 調用、路徑計算與繪製、時間估算、UI 事件
   - js/graph.js圖資料結構（Graph 類），前端本地維護節點與邊的鄰接表
   - js/editor.jsGod Mode 編輯器：節點/邊的新增、刪除、匯出
   - js/lang.js語言與本地化支援（繁中/英文切換）
   - css/styles.css樣式定義（面板、標記、路線卡片等）
   - index.html主頁面結構與 UI 元件定義

【地圖繪畫】
━━━━━━━━━━━━━━━━━━
1.介紹
   包含什麼：在地圖上動態繪製校園地圖要素，包括建築、邊線、路徑
   功能：顯示校園地理信息，支持不同的視覺層（陰影、樹木覆蓋等），實時更新路徑顯示
   怎麼使用：地圖自動加載地理數據，不同層可通過控件切換顯示/隱藏
2.方法論
   技術：Leaflet.js LayerGroup、GeoJSON 渲染、Shapely（Python）用於多邊形計算。底層幾何引擎實作了 EPSG:3826 (TWD97) 投影轉換以確保公尺級別的物理運算精度，並結合 STRtree 空間索引技術，大幅優化大規模路網與陰影多邊形的交集覆蓋率計算效能。
   流程：
   • 後端透過 GeoPandas 讀取預處理過的幾何圖資（ccu_buildings_ready.geojson），作為新版即時陰影 API 動態生成陰影多邊形的運算基底。
   • 根據太陽位置動態計算陰影多邊形
   • 根據導航路徑資料中的 tree_shade 與 building_shade 權重屬性，篩選出具備遮蔽效果的路段，並以線條形式標示於地圖上
   • 將路徑上的建築陰影與樹木覆蓋路段，統一渲染為深色透明的加粗線條以提示使用者
   • 路線計算後立即在地圖上繪製路徑。導航系統內建環境感知邏輯，能根據不同載具類型、真實地形坡度與校園路權限制，動態進行車速折減與「牽車路段」的混合路徑切換，提供高度擬真的時間估算
   語言：JavaScript（Leaflet）+ Python（FastAPI/Shapely）+ C++。後端架構採用 Python (FastAPI) 作為 API 閘道，將高複雜度的圖論與路徑規劃運算委派給專屬的 C++ 核心引擎執行，以確保大規模節點運算的極致效能。
   實現細節：
      • 分別建立 treeShadeLayer 與 buildingShadowLayer 兩個圖層群組（LayerGroup）來獨立管理樹蔭與即時建築陰影的顯示與清除。
      • 用 L.GeoJSON 和自定義 style 函數繪製不同樣式的要素

3.相關檔案：
   js/app.js：層管理和更新邏輯
   index.html：地圖容器和控件
   ccu_buildings_ready.geojson：後端即時陰影多邊形 API 所使用的預處理幾何數據。
   ccu_buildings.geojson：後端陰影計算原始數據
   main.py：API 端點與路徑規劃委派邏輯
   shadow_calculator.py：陰影多邊形生成（含 CRS 轉換與 R-tree 空間索引）

【後端】
━━━━━━━━━━━━━━━━━━
1. 介紹
   - 包含什麼：REST API 服務器，處理路由計算、數據查詢、圖修改請求
   - 功能：接收客戶端請求，調用 C++ 路由引擎計算最優路徑，返回圖數據和陰影信息。具備動態圖資生成與橋接能力，能將 Python 計算出的實時陰影結果，動態轉化為路段權重 (building_shade) 注入至底層圖資，使獨立的 C++ 路由引擎得以支援環境感知的進階路徑規劃。
   - 怎麼使用：啟動 main.py 後會在 localhost:8000 開啟 API 服務，前端通過 HTTP 請求調用

2. 方法論
   - 框架：FastAPI（Python）
   - 流程：
     • 應用啟動時自動清洗高程數據（clean_elevation.py）
     • 初始化陰影計算機（ShadowCalculator）
     • 監聽來自前端的多類核心請求（包含路徑計算、圖數據查詢與儲存、實時多邊形與邊線陰影計算，以及樹木多邊形查詢等）：
      ✓ /api/route：調用 C++ main.exe 計算路徑
      ✓ /api/graph：返回節點和邊的圖數據
      ✓ /api/current-shadow-polygons：計算和返回實時陰影多邊形
      ✓ /api/tree-polygons：返回樹木覆蓋多邊形
     • 支持密碼驗證和圖數據導出功能
   - 語言：Python
   - 實現細節：
     • 使用 lifespan context manager 在伺服器啟動時執行自動化數據清洗與圖資初始化邏輯。
     • 使用 pysolar 計算太陽位置（高度角、方位角）
     • 使用 使用 geopandas 和 shapely 進行地理空間計算。底層幾何引擎實作了 EPSG:3826 (TWD97) 投影轉換以確保公尺級別的物理運算精度，並於記憶體中建立 STRtree 空間索引，大幅優化大規模路網與實時陰影多邊形的交集覆蓋率計算效能。
     • API 內建強健的幾何邊界防護與容錯機制（如限制最低太陽高度角、最大陰影長度上限，及動態清理修復缺失的建築高度屬性），有效避免日落極端角度或圖資髒資料導致的無窮大運算與系統崩潰。
     • CORS 中間件允許跨域請求

3. 相關檔案
   - main.py：FastAPI 應用主文件，API 端點定義
   - shadow_calculator.py：陰影計算引擎
   - clean_elevation.py：高程數據清洗
   - fetch_elevation.py：高程數據獲取
   - requirements.txt：Python 依賴列表

【路由演算法】
━━━━━━━━━━━━━━━━━━
1. 介紹
   - 包含什麼：校園導航的核心路由計算引擎，支持多種路線優化策略
   - 功能：根據起點、終點和路線偏好（最短、最少轉彎、最少爬坡等），計算最優路徑
   - 怎麼使用：前端設置起點、終點、選擇導航模式和路線偏好，後端調用 C++ 引擎計算並返回路線

2. 方法論
   - 核心技術：C++ 實現的路徑計算演算法（負責單段起終點運算），而多目標巡航（TSP）則是由前端 JavaScript 透過貪婪演算法計算直線距離來實作。
   路由引擎底層實作了多維度成本函數 (Multi-factor Cost Function)，能夠無縫接入前置處理的環境數據（如實時建築陰影與樹蔭覆蓋率），將抽象的氣候環境動態轉化為尋路權重，
   達成具備環境感知的智能導航。
   - 流程：
     • 前端發送 POST 請求到 /api/route，包含：starts（起點列表）、ends（終點列表）、mode（導航模式）、vehicle（交通工具）
     • 後端調用 main.exe，傳遞參數：起點ID、終點ID、導航模式、交通工具。系統採用無狀態的微服務解耦架構，
     Python 後端透過子行程 (Subprocess) 與 CLI 參數喚醒 C++ 獨立執行檔，此設計大幅降低了跨語言記憶體管理的複雜度，確保 API 伺服器的高可用性與穩定度。
     • C++ 引擎根據邊的距離、爬坡度、轉彎次數等屬性計算成本。
     在圖搜索過程中實作了動態路網剪枝 (Dynamic Edge Pruning) 機制，引擎會根據傳入的交通工具類型（如步行、單車、機車）即時過濾不具備通行路權的路段，
     確保生成的導航路線符合真實世界的物理與校園規範。
     • 返回最優路徑的節點序列
   - 語言：C++（核心算法）
   - 支持的模式：
      ✓ 距離最短（shortest）
      ✓ 最少轉彎（least_turns）
      ✓ 不流汗路線（least_climbing）
      ✓ 避暑模式（shade）
   - 支持的交通工具：步行（walk）、自行車（bike）等

3. 相關檔案
   - main.exe：編譯後的 C++ 路由引擎
   - main.cpp：C++ 源代碼
   - campus_nodes_edges.json：圖的節點和邊的數據
   - main.py：路由 API 端點（/api/route）

【資料儲存】
━━━━━━━━━━━━━━━━━━
1. 介紹
   - 包含什麼：校園地理數據和圖數據的存儲結構，包括節點、邊、建築、樹木等
   - 功能：持久化存儲校園地圖信息，支持讀取、修改、導出功能
   - 怎麼使用：系統自動讀取 JSON 和 GeoJSON 文件，支持通過 API 修改並保存

2. 方法論
   - 格式：
      • JSON：圖數據（節點、邊及其屬性）
      • GeoJSON：地理空間數據（建築、樹木、陰影）
      • TXT：C++ 路由引擎執行時所需的圖結構輸入檔案（Input Format）。
      TXT 檔案並非僅用於除錯，而是系統的「圖資編譯產物」。
      系統以 JSON 為核心真相來源，動態生成扁平化、包含多維度權重 (如路權、實時陰影) 的 TXT 檔案，作為 Python API 與 C++ 路由引擎間的高效跨語言中介橋樑。
   - 流程：
      • 實作了空間資料預處理管線 (ETL Pipeline)，透過專屬腳本將原始圖資預先轉換為優化後的就緒檔 (_ready.geojson)，將耗時的幾何清理與屬性標準化從 API 執行期轉移至建置期，顯著提升伺服器啟動速度與運算效能
      • 系統啟動時讀取 campus_nodes_edges.json（圖的唯一真相來源）
      • 讀取 ccu_buildings.geojson 和 ccu_trees_ready.geojson（地理數據）
      • 根據數據初始化陰影計算機。資料載入層內建自動修復機制 (Auto-healing Mechanism)，能在系統啟動時主動偵測並補齊圖資中缺失的拓樸屬性 (如自動補全 Edge ID 並持久化)，確保圖論演算法的執行安全與系統高可用性
      • 支持通過 POST /api/graph 修改並保存圖數據
   - 語言：JSON、Python（json 模塊）、JavaScript（fetch API）
   
3. 相關檔案
   - campus_nodes_edges.json：圖的節點和邊（核心數據文件）
   - campus_nodes_edges_categorized.json：已包含分類屬性的圖數據檔案
   - campus_nodes_edges_backup_[時間戳記].json：系統修改前自動生成的圖數據備份文件
   - ccu_buildings.geojson：建築地理數據
   - ccu_buildings_ready.geojson：處理後的建築數據
   - ccu_trees_ready.geojson：樹木地理數據
   - graph.txt：圖的文本表示
   - nodes.txt：節點數據
   - prepare_data.py：專案內含之輔助資料轉換腳本（但非系統核心啟動流程之必要組件）

【導航模式】

【最短路徑（Shortest Path）】
1.介紹
   包含什麼：計算起點到終點距離最短的路徑。支援多目的地巡航導航，並將多節點排序的運算壓力卸載至客戶端，利用前端貪婪演算法預先進行空間距離最佳化排序後，再分段請求精確路由，實現高效的邊緣運算架構。
   功能：在校園內快速導航到目的地，優先考慮距離而非其他因素。系統不僅計算最短距離，前端更內建了物理估算引擎，能根據真實路段坡度與不同交通工具的特性，
   動態折減行駛速度，提供高度擬真的預估旅行時間 (ETA)。
   怎麼使用：選擇起點、終點，在路線偏好中選擇「距離最短」，點擊「開始巡航導航」。

2.方法論
   技術：Dijkstra 算法，邊權重為實際距離（單位：米）
   流程：
   • 前端設置 mode="shortest"
   • C++ 引擎計算圖中起點到終點的最短距離路徑
   • 沿著經過的邊累加距離，找到最小值
   • 前端地圖渲染引擎具備路權感知能力，能針對特定載具（如機車禁行區）自動切分路線圖層，無縫切換為「牽車步行」的虛線視覺化呈現，並提供動態的校園法規警示。
   實現：main.cpp 中的 Dijkstra 實現，以距離作為成本函數

3.相關檔案
   main.cpp：最短路徑算法實現
   main.py：API 路由（將 "shortest" 作為 mode 屬性，亦即傳遞給 C++ 引擎的第三個參數）
   app.js：前端地圖渲染與邊緣運算核心。負責處理 Leaflet 視覺化交互、執行多目標巡航的 TSP 貪婪預排序、動態計算考量地形坡度的旅行時間 (ETA)，以及依據路權動態渲染「牽車步行」的視覺線段。

【依順序巡航（Sequential）】
1.介紹
   包含什麼：順序訪問多個目的地，按照給定的順序依次到達
   功能：適合需要按特定順序訪問多個地點的場景（如巡檢、送貨）。
   前端渲染引擎將路徑拓樸資料與途經點業務邏輯解耦，針對多目標巡航動態生成帶有次序編號與狀態顏色的視覺化標記 (Sequence Markers)，大幅提升連續導航場景的使用者體驗。
   怎麼使用：輸入多個終點，選擇「依順序巡航」模式，系統會按順序計算路徑

2.方法論
   技術：依次計算相鄰兩個節點間的最短路徑，串接成完整路線。系統採用前端編排架構 (Frontend Orchestration)，由客戶端將連續巡航任務拆解為多個獨立的點對點 API 請求，
   維持了後端運算引擎的無狀態性，並有效降低單一請求的逾時風險。

   流程：
   • 前端在「依順序巡航」模式下，會透過迴圈將多個目的地拆解，逐次發送單段路徑的 POST 請求至 /api/route，請求中包含單次計算的起點與終點
   • 前端 JavaScript 依次向後端發起單段路徑請求（starts[0]→ends[0]、ends[0]→ends[1]...），而 C++ 引擎僅負責接收單次請求並回傳單段的計算結果
   • 將所有路段串接成一條完整路線

   實現：前端透過迴圈多次請求 API，並在接收到各段路徑結果後，於 JavaScript 中將陣列拼接成一條完整的巡航路線。在客戶端實作了精準的無縫路徑拼接邏輯，
   透過陣列切片 (Slice) 去除微段路徑交接處的重複節點，確保最終渲染的全局路徑拓樸具備絕對的連續性與資料正確性。

3.相關檔案
   app.js：前端編排核心。負責拆解依序巡航任務、發送點對點請求、透過陣列切片 (slice) 拼接無縫路徑，以及渲染帶有次序編號與狀態顏色的視覺化標記 (Sequence Markers)。
   main.cpp：負責單段起終點路徑計算（無 sequential 模式實作）
   main.py：API 路由

【自動規劃多目的地 TSP（Traveling Salesman Problem）】

1.介紹
   包含什麼：自動找到訪問多個目的地且總距離最短的路線
   功能：智能規劃路線順序，適合需要高效訪問多個位置的場景（如外勤工作）
   怎麼使用：輸入起點和多個終點，在巡航策略選擇「多目的地 (自動優化順序-TSP)」模式，系統會自動確定訪問順序

2.方法論
   技術：前端 JavaScript 實作的貪婪演算法。系統實作了邊緣運算架構，將多目標的 TSP 排序運算卸載至前端客戶端，
   利用地理直線距離進行輕量化的貪婪啟發式排序 (Greedy Heuristic)，極大化地減輕了後端伺服器的運算負載。

   流程：
   • 前端採用動態交錯運算策略，在迴圈中逐次算出直線距離最近的下一節點後，便立即向 API 發送該單段的起終點與路線偏好參數進行請求；
   並透過微段請求編排機制，將動態取得的子路徑利用切片去重演算法無縫拼接，確保全局導航拓樸的連續性與系統高可用性。
   • 前端實作了微段請求編排機制，將排序好的巡航節點拆分為獨立的非同步 API 請求，並透過節點去重演算法無縫拼接子路徑，確保全局導航拓樸的連續性與系統高可用性
   • C++ 引擎僅負責接收前端每次傳來的單段起點與終點，計算並返回該單段最優路線的節點序列
   • 巡航引擎內建斷點容錯機制 (Fault Tolerance)，當偵測到局部網格因路權限制或拓樸斷裂導致無法連通時，能自動捨棄該孤島節點並接續規劃後續行程，確保整體導航系統的強健性

   實現：透過計算節點間的地理直線距離 (getLinearDist)，逐次選擇最近的下一個目的地

相關檔案
   app.js：前端 TSP 排序、微段請求編排與斷點容錯機制實作
   main.cpp：負責執行 Dijkstra 演算法計算單段路線
   main.py：API 路由

【最短路徑（Shortest Distance）】
1.介紹
   包含什麼：計算起點到終點距離最短的路徑。支援多目的地巡航導航，並將多節點排序的運算壓力卸載至客戶端，利用前端貪婪演算法預先進行空間距離最佳化排序後，再分段請求精確路由，實現高效的邊緣運算架構。
   功能：在校園內快速導航到目的地，優先考慮距離而非其他因素。系統不僅計算最短距離，前端更內建了物理估算引擎，能根據真實路段坡度與不同交通工具的特性，
   動態折減行駛速度，提供高度擬真的預估旅行時間 (ETA)。
   怎麼使用：選擇起點、終點，在路線偏好中選擇「距離最短」，點擊「開始巡航導航」。

2.方法論
   技術：Dijkstra 算法，邊權重為實際距離（單位：米）
   流程：
   • 前端設置 mode="shortest"
   • C++ 引擎計算圖中起點到終點的最短距離路徑
   • 沿著經過的邊累加距離，找到最小值
   • 前端地圖渲染引擎具備路權感知能力，能針對特定載具（如機車禁行區）自動切分路線圖層，無縫切換為「牽車步行」的虛線視覺化呈現，並提供動態的校園法規警示。
   實現：main.cpp 中的 Dijkstra 實現，以距離作為成本函數

3.相關檔案
   main.cpp：最短路徑算法實現
   main.py：API 路由（將 "shortest" 作為 mode 屬性，亦即傳遞給 C++ 引擎的第三個參數）
   app.js：前端地圖渲染與邊緣運算核心。負責處理 Leaflet 視覺化交互、執行多目標巡航的 TSP 貪婪預排序、動態計算考量地形坡度的旅行時間 (ETA)，以及依據路權動態渲染「牽車步行」的視覺線段。

【最少轉彎（Fewest Turns）】

1.介紹
   包含什麼：優先選擇轉彎次數最少的路線
   功能：提高導航體驗，減少複雜的轉向指示
   怎麼使用：在路線偏好中選擇「最少轉彎」

2.方法論
   技術：在 Dijkstra 演算法執行時，透過節點（Nodes）的經緯度座標，動態計算行進向量的夾角來作為額外的轉彎懲罰成本。突破標準 Dijkstra 演算法的狀態限制，
   實作了路徑相依的狀態空間展開機制，在圖搜索過程中動態追蹤前驅節點 (Previous Node)，以精確處理非馬可夫屬性的連續轉彎計算。
   流程：
   • 捨棄靜態的預設邊權重，C++ 核心引擎透過載入拓樸節點的真實空間座標，於搜索遍歷時動態執行向量幾何運算，精準計算出相鄰路段的真實夾角與轉彎程度
   • C++ 算法優先選擇轉彎少的邊
   實現：在 Dijkstra 的成本函數中加入轉彎懲罰。實作了複合式成本函數，將幾何轉彎角度量化為動態的「距離懲罰權重 (Turn Penalty)」並疊加於物理距離之上，
   有效避免演算法為追求極致的少轉彎而規劃出過度繞行的無效路線。

3.相關檔案
   nodes.txt：由系統動態生成的扁平化節點座標檔，供 C++ 引擎載入並於尋路時動態計算轉彎角度。
   main.cpp：轉彎成本計算

【最少爬坡（Least Elevation Gain）】

1.介紹
   包含什麼：優先選擇爬坡度最小的路線
   功能：減輕體力消耗，適合行動不便或不想登山的使用者。將高程資料的應用延伸至前端物理引擎，能根據真實坡度與載具特性套用動態折速公式（如遇陡坡自動將車速降至步行速度），
   並配合圖資規範的路權狀態提供極致擬真的導航時間估算。
   怎麼使用：在路線偏好中選擇「不流汗路線」

2.方法論
   技術：
   • 邊權重為爬坡度或海拔增益。路由引擎底層實作了非對稱向性的坡度成本模型，能精準判別「上坡懲罰」與「下坡豁免」的單向物理差異，確保演算法能正確反映真實世界中不同行進方向的體力消耗。
   • 數據來源：數據來源：Open-Elevation API 提供的開源高程數據

   流程：
   • 於系統啟動時透過 clean_elevation.py 自動清洗並載入高程圖資
   • 根據相鄰節點的海拔差計算邊的爬坡度：slope = (elevation[to] - elevation[from]) / distance
   • C++ 算法優先選擇爬坡度小的邊

   實現：
   • fetch_elevation.py：批量查詢海拔（OpenTopography API）
   • clean_elevation.py：導入了專屬的高程圖資清洗管線，主動過濾並平滑化因衛星雷達或 API 誤差產生的異常海拔突變 (Outliers)，防止微觀地形雜訊干擾巨觀的路徑規劃演算法。
   • main.cpp：爬坡成本函數

3.相關檔案
   fetch_elevation.py：海拔數據獲取
   campus_nodes_edges.json：節點的海拔和邊的爬坡度
   main.cpp：爬坡成本計算
   clean_elevation.py：高程數據清洗(因海拔數據的解析度是30公尺X30公尺，會造成部分edge的slope是懸崖的程度)

【日曬地圖（Shade/Sun Map）】

1.介紹
   包含什麼：顯示當前時間校園各位置的陰影覆蓋情況。系統不僅動態計算建築陰影，更利用影像處理與電腦視覺技術 (analyze_shade.py)，預先從空拍圖中產生樹木特徵遮罩，並透過 Bresenham 直線演算法直接計算路徑圖的像素覆蓋率來量化樹蔭權重，將靜態樹蔭與動態建築陰影深度疊加，構建出高保真度的複合式避暑環境圖資。
   功能：使用者可以查看日曬情況，選擇避日的路線
   怎麼使用：使用者規劃路線後，可手動開啟陰影顯示開關，系統會在當前導航路徑上標示出具備遮蔽效果的路段，選擇路線時可考慮陰影覆蓋

2.方法論
   技術：
   • Pysolar：計算太陽位置（高度角、方位角）
   • Shapely：計算建築投影陰影。為滿足實時路網權重更新的效能需求，底層陰影計算引擎導入了 STRtree 空間索引技術，大幅降低了不規則陰影多邊形與龐大路網線段進行交集覆蓋率運算時的時間複雜度。
   • GeoJSON：存儲和渲染陰影多邊形

   流程：
   • 系統啟動時載入建築 GeoJSON（ccu_buildings.geojson）
   • 陰影多邊形計算（如 /api/current-shadow-polygons 端點）：
   ✓ 計算當前時間的太陽位置（高度角 altitude、方位角 azimuth）
   ✓ 根據建築高度和太陽位置計算投影方向
   ✓ 使用 Shapely 計算建築在地面上的投影陰影
   ✓ 在生成陰影多邊形時，系統將建築物原地幾何與依據太陽角度平移後的投影幾何進行聯集，並透過運算凸包 (Convex Hull) 快速且有效地包絡出完整的實體陰影範圍。
   • 前端渲染：前端直接讀取路徑線段（Edge）中的 tree_shade 與 building_shade 屬性，篩選出陰影覆蓋的路段，並以深色透明線條疊加渲染於路徑上

   實現細節：
   • 選擇「避暑/林蔭模式」路線偏好時，C++ 算法將邊的「陰影覆蓋度」作為成本
   • 優先選擇被陰影覆蓋較多的邊

   語言：Python（Pysolar、Shapely）+ JavaScript（渲染）

3.相關檔案
   main.py：/api/current-shadow-polygons 端點
   shadow_calculator.py：陰影多邊形計算引擎
   ccu_buildings.geojson：建築數據（包括高度）
   js/app.js：陰影層渲染和更新
   main.cpp：陰影成本計算（可能有 shadow 相關屬性）
   analyze_shade.py：萃取校園樹冠多邊形之影像處理腳本
